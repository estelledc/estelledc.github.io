On this case

问题系统关键取舍证据边界

零基础用户缺的不是命令列表，而是一条不会突然断掉的完成路径。

工具文档通常从功能出发，却默认读者已经懂终端、文件、Git 和发布。当任意一个前置概念缺失，教程就会变成“照着输入，但不知道发生了什么”。

Zero to AI 把完成标准改成公开工件：用户需要能解释改了什么，能运行检查，能保留 Git 历史，并把一个小项目发布为可访问的页面。

两条工具路线，共享一套工程基本功。

  1. Claude Code 与 Codex 各有独立入口、环境说明和第一次只读任务。
  2. 修改、验证、Git 与发布方法放入共享板块，避免重复和冲突。
  3. 每篇遵循日常类比、技术定义、实际操作、常见坑、自检和下一步。
  4. 学习路径状态保存在本地浏览器，并区分“访问过”与“主动完成”。
  5. 交叉引用、官方来源、时效性、fixture、构建和端到端 smoke 共同守住发布。

教学产品的“功能”是降低失控感。

先只读，再授权修改

第一项任务要求用户观察项目而不是立即让 AI 改代码，先建立“工具会读什么、我能检查什么”的安全直觉。

完成必须留下证据

最终作品要求公开 Pages URL、Git 历史、验收截图和项目规则，避免把“AI 说完成了”当成完成。

时效性是一等内容字段

版本相关页面记录最后核验日期与工具版本；官方来源矩阵和审计脚本负责提醒内容过期，而不是依赖作者记忆。

边界与延期证据。

Verification boundary

自动检查能验证教程结构、链接、fixture、构建和浏览器交互，但不能代替真实账户权限、方案额度、组织策略或不同操作系统上的完整迁移测试。

工具界面与命令会变化。站点通过官方来源与核验日期降低过期风险，但不承诺教程永久适用于未来版本；读者遇到差异时应以工具官方文档为准。

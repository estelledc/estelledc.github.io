This case answers

说说你如何设计一个每天只出现一张纸条的产品，而不把它变成需要账号、云端和无限 Feed 的另一个内容应用。

产品边界是设计本身：本地资源决定每日纸条，涂鸦、分享和 Widget 都围绕这个有限仪式工作。

On this case

问题系统关键取舍证据边界

不是再造一个内容 Feed，而是让一天只出现一张纸条。

小柴把信息量主动压到最低：每天由本地资源决定一张纸条，用户可以读、画、分享，也可以什么都不做。最近 14 天的导航提供回看，但不制造无限下滑。

产品的信任前提同样简单：纸条、涂鸦和偏好不需要账户，不上传云端，也不依赖第三方分析服务。

UIKit 主应用与 WidgetKit 共用一个确定性的本地模型。

  1. Bundled JSON 提供 365 条常规纸条与 1 条闰日特例，本地文件保存用户涂鸦与偏好。
  2. UIKit 纯代码布局承载纸条、近 14 天日期、全屏涂鸦、撤销 / 重做、整笔橡皮、可恢复清空和主题交互。
  3. 分享面板实时预览竖版或方形卡片，确认后再生成高清图片并交给系统分享面板。
  4. 用户可以通过系统 Files provider 主动导出或导入版本化本地备份；导入先校验，再原子合并。
  5. WidgetKit 读取 App Group payload；共享容器不可用时显示 bundled deterministic fallback。
  6. 版本、隐私清单、发布标识、图标、iPhone / iPad 构建与 UI 测试进入统一 Makefile 门禁。

隐私不是说明文字，而是架构中没有网络层。

主 App 使用 UIKit，Widget 使用 SwiftUI

UIKit 负责需要手势、绘制和生命周期控制的主界面；WidgetKit 的 SwiftUI 声明式视图负责系统小组件。两种 UI 技术围绕共享模型协作，而不是强行统一。

共享失败时不悄悄写回普通偏好

App Group suite 不可用时，主应用不会回退到标准 UserDefaults；Widget 只展示可预测的 bundled fallback，避免两个进程看起来同步、实际数据分叉。

发布检查前置

App、Widget 与 Tests 的版本一致性、Privacy Manifest、标识和图标由脚本检查，减少“模拟器能跑但发布配置缺失”的后期返工。
